GitHub Actions: The Pragmatic Choice for Most Teams

GitHub Actions won the CI/CD wars for a simple reason: it removed the biggest friction in CI—setting up and maintaining your own runner infrastructure. For most teams, GitHub-hosted runners are enough. You write YAML, GitHub handles the rest.

GitHub Actions vs Jenkins: The Unpopular Opinion

Here's a take you'll rarely see written down:  Jenkins's flexibility used to be its strength; now that same flexibility creates maintenance burden. Plugin version conflicts, JVM tuning, master-agent architecture overhead—these are real costs that compound as your team grows.

That said, Jenkins still wins in two scenarios: (1) you have a massive monorepo with complex matrix builds that GitHub Actions' billing model makes expensive, or (2) you have deep existing Jenkins infrastructure and the migration cost exceeds the maintenance cost. If neither applies, move to GitHub Actions.

Containers and Kubernetes: The Runtime Layer

Multi-Stage Dockerfile: Production-Grade Images

dockerfile

# Stage 1: Build

FROM maven:3.9-eclipse-temurin-17 AS builder

WORKDIR /app

# Download dependencies first (layer caching optimization)

COPY pom.xml .

RUN mvn dependency:go-offline -B

COPY src ./src

RUN mvn clean package -B -DskipTests

# Stage 2: Runtime

FROM eclipse-temurin:17-jre-alpine AS runtime

# Security: run as non-root

RUN addgroup -S appgroup && adduser -S appuser -G appgroup

WORKDIR /app

# Copy only the artifact, not the entire build context

COPY --from=builder --chown=appuser:appgroup /app/target/myapp.jar app.jar

# Health check

HEALTHCHECK --interval=30s --timeout=5s --start-period=30s --retries=3 \

CMD wget -qO- http://localhost:8080/actuator/health/liveness || exit 1

USER appuser

EXPOSE 8080

# JVM tuning for container awareness

ENTRYPOINT ["sh", "-c", "java \

-XX:+UseContainerSupport \

-XX:MaxRAMPercentage=75.0 \

-XX:+UseG1GC \

-jar app.jar"]

# prometheus-rule-example.yaml

apiVersion: monitoring.coreos.com/v1

kind: PrometheusRule

metadata:

name: myapp-alerts

namespace: monitoring

spec:

groups:

- name: myappavailability

rules:

- alert: HighErrorRate

expr: |

sum(rate(http_server_requests_seconds_count{

status=~"5..", job="myapp-api"

}[5m])) / sum(rate(http_server_requests_seconds_count{

job="myapp-api"

}[5m])) > 0.05

for: 2m

labels:

severity: critical

team: platform

annotations:

summary: "High error rate detected on myapp-api"

description: "Error rate is {{ $value | humanizePercentage }} (threshold: 5%)"

runbook_url: "https://runbooks.mycompany.com/high-error-rate"

- alert: PodMemoryUsageHigh

expr: |

(sum(container_memory_working_set_bytes{

pod=~"myapp-api-.*", namespace="production"

}) by (pod) / sum(container_spec_memory_limit_bytes{

pod=~"myapp-api-.*", namespace="production"

}) by (pod)) > 0.85

for: 5m

labels:

severity: warning

annotations:

summary: "Pod {{ $labels.pod }} memory usage above 85%"

 Every alert you define should correspond to a runbook. If you can't write a runbook for an alert, you don't understand the alert well enough to page someone at 3 AM.

The key insight:  Code commits trigger Actions, which build images and store them in a registry, which Argo CD detects and syncs to Kubernetes, which Prometheus monitors and alerts on. The pipeline doesn't end at deployment—it closes at observability.

DevOps automation at this level isn't cheap to set up—expect 3-6 months for a team of 3-5 to reach production-grade maturity. But the compounding returns are real: deployments go from stressful all-hands events to routine, reliable operations that require minimal human intervention. That's the actual goal—not automation for its own sake, but the freedom it creates for your team to focus on building product instead of fighting infrastructure.
